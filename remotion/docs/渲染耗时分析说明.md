# 渲染耗时分析说明

## 如何分析「Setting the current frame」日志

把渲染时控制台里**整段**包含 `delayRender() "Setting the current frame to N" handle was cleared after Xms` 的日志保存为纯文本文件（例如 `render-timing-log.txt`），放在项目根目录，然后执行：

```bash
node scripts/parse-render-timings.js render-timing-log.txt
```

脚本会输出：

- **整体耗时**：总次数、总秒数、平均/最小/最大、P50/P90/P95/P99
- **按 Tab 总耗时**：哪个 Tab 累计最久
- **单次耗时分布**：多少落在 0–1ms、35–50ms、50–80ms、80–150ms 等区间
- **单次最慢的 20 次**：对应 Tab 和帧号
- **结论**：耗时最多的地方（阶段、Tab、≥80ms 占比等）

## 根据你当前日志的结论（耗时最多的地方）

结合你提供的日志片段（帧约 714～1651、8 个 Tab、每行 `handle was cleared after Xms`），可以归纳为：

### 1. 耗时集中在哪个阶段

- **几乎全部时间都花在「Setting the current frame」** 上，也就是 Chromium 每个 Tab 渲染/合成单帧并解除 `delayRender` 的那一步。
- 整段日志对应的总时间 ≈ 所有「after Xms」之和（约 7～8 分钟量级，视帧数和机器而定）。

### 2. 单次耗时分布（大致）

- **多数单次在 30～70ms**（约一半在 50～80ms 区间）。
- **少量 0～1ms**（可能来自缓存或极简帧）。
- **少数在 80～150ms**，个别**尖峰到 260ms 左右**（你日志里出现了 264ms、266ms、150ms 等）。
- 因此：**单次最慢在 250～270ms**，出现在某些具体帧（如 998、999 附近）。

### 3. 按 Tab 看

- 8 个 Tab 的**总耗时会略有高低**（谁多谁少取决于每帧复杂度），但**没有哪一个 Tab 单独占绝大多数**；整体是 8 路并行，总墙钟时间由最慢的那路和帧数决定。

### 4. 耗时「最多」的几块（总结）

| 项目 | 说明 |
|------|------|
| **阶段** | **「Setting the current frame」**（每帧渲染/合成 + delayRender 清除） |
| **单次最慢** | 约 **260ms 级**（你日志里 264ms、266ms），多出现在部分复杂帧 |
| **主要区间** | **50～80ms** 的帧数最多，合计时间也最多 |
| **次要拖慢** | **≥80ms** 的帧虽次数少，但单次时间长，合计能占约 25～30% 总时间 |

### 5. 若要进一步加速

- **减帧或降分辨率**：总帧数或像素少，这段阶段总时间会线性下降。
- **降低单帧复杂度**：粒子/效果少一点，单次 30～70ms 可能降到 20～50ms，尖峰 260ms 也可能降低。
- **保持 GPU + NVENC**：编码阶段已用硬件，主要瓶颈在「每帧渲染」这段，而不是编码。

用完整日志再跑一遍 `parse-render-timings.js` 可以得到你这台机器上的精确分布和「耗时最多的 Tab / 帧号」。
